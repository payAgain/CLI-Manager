# 优化会话历史列表与搜索性能

## Goal

将历史列表与全文搜索从同步扫描 JSONL 改为缓存优先的持久化索引，避免当前 1466 个文件、约 854MB 日志规模下的秒级等待。

## Changelog Target

`[TEMP]`

## Requirements

- 使用独立、可重建的 SQLite 缓存库保存会话摘要和 FTS5 消息索引，不修改主业务数据库，不新增依赖。
- 列表优先返回缓存，后台按文件指纹增量同步；项目筛选不得在请求路径逐个打开 JSONL。
- 全文搜索至少输入 3 个 Unicode 字符，使用 FTS5 trigram，保留三字符以上任意子串匹配。
- 首次建索引期间显示进度和部分结果，完成后自动刷新当前列表与查询。
- 保留现有来源/项目过滤、分页、收藏、删除、详情打开和实时会话精确查询行为。
- 前端必须防止旧搜索请求覆盖新结果，新增文案同时支持 zh-CN 与 en-US。

## Acceptance Criteria

- [ ] 缓存列表与三字符以上搜索在当前数据规模下 P95 不超过 200ms。
- [ ] `history_list_sessions` 和 `history_search` 的现有输入及数组结果保持兼容。
- [ ] 未变化文件不重解析，新增/变化/删除文件只更新对应索引记录。
- [ ] 索引失败或损坏时保留旧结果或自动重建，不影响源 JSONL。
- [ ] 首次索引期间可搜索已完成部分，完成后自动补全结果。
- [ ] `cargo test history`、`cargo check`、`npx tsc --noEmit` 通过。

## Out of Scope

- 重构历史统计聚合缓存和会话详情解析。
- 修改搜索结果排序、收藏语义或历史消息展示结构。
- 新增依赖、后台常驻服务或主数据库迁移。
