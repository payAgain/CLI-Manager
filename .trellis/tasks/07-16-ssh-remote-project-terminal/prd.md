# 设计 SSH 远程项目与远程终端

## Goal

为 CLI-Manager 设计并分阶段实现 SSH 远程项目与远程终端，使“本机作为入口、开发环境位于远程 Linux 主机”的工作流成为一等公民，同时保持现有本地项目、Worktree、终端分屏和 AI CLI 工作流的兼容性。

本任务必须从全应用架构和完整功能关联出发，不采用“用户提出一个字段就局部追加一个字段”的设计方式。设计先建立现有能力地图、路径依赖和跨功能边界，再确定 SSH 产品模型与分期方案。

关联 Issue：https://github.com/dark-hxx/CLI-Manager/issues/145

Changelog Target: `V1.3.0`

## Confirmed Facts

- 当前项目模型以本地路径为核心，终端由本地 PTY 启动；手动执行 `ssh` 后，项目树、路径检查、Git、文件浏览、会话绑定和统计仍按本地路径工作。
- Issue 建议 MVP 聚焦“远程项目配置 + SSH 终端会话”，不在首期实现完整远程 IDE。
- SSH 实现优先薄封装系统 OpenSSH，复用用户已有的 `~/.ssh/config`、SSH Agent、ProxyJump、硬件密钥和主机指纹机制。
- 一台 SSH 主机需要能够承载多个远程项目目录；“SSH 主机”和“远程项目”不应被建模为同一个实体。
- 用户期望按“设计方案 → UI 出图 → 执行文档 → 进入实现”的顺序推进。
- SSH 主机统一在“设置 → SSH 主机”中管理；新建项目时选择“本地项目 / SSH 远程项目”，远程项目再选择 SSH 主机和远程目录。
- 项目列表保持现有结构，不自动按 SSH 主机建立系统分组，也不增加“平铺/主机视图”切换；用户可使用现有手动分组组织本地与远程项目。
- 新建/编辑项目保持现有“新增终端”窄幅单列表单的视觉和交互结构；SSH 远程项目作为现有表单中的运行环境分支，仅增加必要的主机与远程路径字段。
- “新增终端”弹窗顶部必须增加类型 Tab，固定为“本地项目”和“SSH 远程项目”。默认选中“本地项目”；本地 Tab 对应并完整保留现有项目表单，SSH Tab 对应新的 SSH 纵向表单。类型切换不得丢失用户在当前类型中已填写但尚未提交的内容。

## Requirements

- 建立 SSH 主机配置与远程项目配置的产品模型、数据边界和生命周期。
- 支持纯 SSH 远程终端，以及绑定远程工作目录的 SSH 远程项目。
- 设计服务器管理入口、新建/编辑项目流程、项目树展示与远程状态表达。
- 现有项目表单需根据 `local / wsl / ssh` 运行环境切换专属字段：本地继续显示路径、Shell 和 Worktree；SSH 显示主机、远程路径和能力提示，并隐藏不适用的本地字段。
- 类型 Tab 是表单一级导航，不使用普通下拉框替代；切换时标题、底部取消/新增操作和整体弹窗尺寸保持稳定，表单内容区独立滚动。
- SSH 主机连接档案需覆盖：名称、手动分组、地址、端口、验证方式、登录用户、认证材料引用、认证顺序、跳板机、代理、备注、连接超时和初始化行为。
- 验证方式设计至少评估：密码、私钥、复用登录凭证、交互认证、每次询问；不得默认明文保存密码或私钥口令。
- 跳板机与代理作为高级连接能力设计，支持清晰的启用状态、引用已有主机配置和连接链路校验。
- 新建 SSH 远程项目时需支持远程目录选择，而不只是让用户盲填路径；目录选择器必须先建立经过认证的临时 SSH 连接，并支持目录浏览、手工输入、路径检测和重试。
- 远程目录浏览能力应与普通终端会话解耦：它是短生命周期的结构化查询会话，不应在项目创建前生成正式终端标签或污染会话历史。
- 制定本地项目、远程项目、Worktree、分屏、Workspan、历史会话、Git、文件浏览、Hook、统计等能力的兼容边界。
- 制定认证、主机指纹、命令转义、凭据存储、断线和错误恢复的安全边界。
- 对比 XTerminal、Termius、Tabby、WindTerm、VS Code Remote SSH、JetBrains Gateway，以及 Issue 提到的 Orca、cmux。
- 所有新增用户可见文案必须覆盖 `zh-CN` 与 `en-US`，且英文模式继续使用 24 小时制。
- 首期优先支持 Windows 本机 OpenSSH；设计需预留 macOS/Linux 兼容，不假定只有 PowerShell 环境。

## Proposed Scope Phases

- P0 调研与设计：形成上述规划文档和 UI 方案，不改产品代码。
- P1 SSH 基础与 MVP：SSH 主机管理、远程项目、系统 SSH 终端、远程标识和基础连接诊断。
- P2 远程项目增强：远程目录检查、远程 Git、文件浏览等按独立能力逐项启用。
- P3 AI 工作流联动：远端 Claude/Codex 历史、Hook、统计、恢复与高级会话能力。

## Out of Scope for MVP

- 自研 SSH 协议栈或替代系统 OpenSSH。
- 内置私钥生成、SSH Agent 或完整密码保险箱。
- 首期即实现 VS Code Remote SSH 级别的远程 IDE、端口转发、SFTP 文件管理和远程扩展系统。
- 未经能力探测就让远程项目假装支持全部本地项目功能。

## Open Questions

- P1 已纳入系统凭据库保存 SSH 登录密码：SQLite 仅保存 `credential_ref`，实际密码进入 Windows Credential Manager、macOS Keychain 或 Linux Secret Service，并通过一次性 AskPass 通道交给系统 OpenSSH。
- UI 审阅阶段确认 SSH 配置表单的信息密度、分区方式与连接诊断呈现。
- P2/P3 的远程文件、Git、历史、Hook 优先级在 MVP 设计通过后单独排序，不在 P1 中隐式承诺。
