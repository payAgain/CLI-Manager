# OSC 10/11 颜色探测研究

## 现象

Codex 启动时查询终端默认前景色和背景色。若完整响应没有在短探测窗口内被探测代码消费，响应会落入正常输入事件循环，显示为 `]10;rgb:...` 和 `]11;rgb:...`。

## 上游证据

- openai/codex#4759、#4945、#5107 均记录了同类 OSC 响应进入 composer 的问题。
- openai/codex#5935 曾通过协调 crossterm 颜色查询与输入循环修复同类竞争。
- openai/codex#23489 指出当前启动颜色探测仍使用约 100ms 固定窗口，SSH、代理和异步终端链路不可靠。
- Codex Windows 颜色探测实现会批量写入 OSC 10/11 查询，要求同时收到两个完整回复才成功。

## CLI-Manager 约束

- 当前查询从 PTY 经 daemon/WebSocket 到前端，再通过反向链路写回 PTY，存在不必要的往返和调度点。
- Replay 与实时输出共用规范化入口，但现有代码已通过 `replyToColorQueries` 区分；迁移后前端应彻底无回复副作用。
- PTY 字节块没有控制序列边界保证，后端解析必须保留跨块状态。
- PTY 输入必须保持单 writer 顺序，不能由 reader 线程和客户端写入并发写同一底层句柄。

## 可行方案

### A. 禁止回复

- 优点：绝对不会产生响应泄漏。
- 缺点：Codex 无法获得真实终端颜色，回退主题可能错误。

### B. 前端继续回复并优化调度

- 优点：改动较小。
- 缺点：无法消除 WebView、WebSocket、daemon 和 PTY 往返，仍是概率性修复。

### C. Rust PTY 层即时回复（已选择）

- 优点：延迟最低；跨分片解析和 Replay 隔离可单元测试；前端不再向输入流注入终端响应。
- 缺点：需要扩展会话颜色协议，并调整 writer 所有权。

## 结论

本地 PTY 应采用方案 C。对 SSH/WSL，需要明确是否以“绝不污染输入”为优先，必要时禁止回复，因为网络 RTT 不受本地后端控制。
