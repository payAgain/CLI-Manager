# Design

## History Catalog

- Add one process-wide async mutex around `ensure_schema` in `open_catalog_once`.
- Open connections may proceed concurrently, but only one connection performs version checks and compatibility DDL at a time.
- Keep the existing column helper and scoped-index DDL; write `PRAGMA user_version` after `history_meta` updates.

## WebDAV Database Restore

- The frontend normalizes backup rows exactly as today and builds parameterized `DELETE`/`INSERT` statements.
- A new `backup_restore_database` Tauri command accepts those statements, validates statement shape/table ownership, opens `.cli-manager/cli-manager.db`, and executes all statements inside one `BEGIN IMMEDIATE` transaction on one SQLx connection.
- The command uses WAL, foreign keys, and a 15-second busy timeout. Any validation or SQL failure rolls back before returning an error.
- Preferences, notifications, and statusline remain in their existing stores/files and execute only after the database transaction commits.

## Compatibility

- No snapshot format or existing command payload changes.
- Existing safety snapshots and automatic rollback call the same fixed restore path.
- The new command is additive and accepts only database-domain statements generated by CLI-Manager.

## SSH Persistence

- Add explicit Rust commands for group schema compatibility, Config host import, host/group deletion, host tool preferences, and Hook report persistence.
- Each combination write opens one WAL connection with a 15-second busy timeout and holds `BEGIN IMMEDIATE` only around its dependent reads and writes.
- Only schema DDL/backfill uses a process-wide single-flight lock plus an atomic success fast path. Ordinary SSH CRUD has no application mutex.
- Hook report selection, retained-root replacement, insertion, and mirror updates execute in one transaction so the decision cannot race its writes.
- Config import loads the existing normalized alias set once, then inserts missing hosts in the same transaction.

## Remote History Concurrency

- Frontend requests with identical launch/source/root/project/cursor/limit/scope inputs share one Promise, including integration metadata persistence.
- Catalog apply first obtains SQLite's writer lease, then compares persisted generation and `generation:offset` cursor state.
- Lower generations and lower offsets in the same generation return `applied=false`; the frontend keeps its current context.
- No global history refresh or SSH mutex is used. Different requests run independently until SQLite's unavoidable short writer section.

## Rollback

- Reverting the frontend to direct plugin SQL restores old behavior; no schema or stored data format changes are introduced.
