# 历史来源接入规划

## 结论

历史来源必须从 Hook 配置中拆出来，独立放到 `设置 -> 历史来源`。

现有 `file_path` 思路需要升级为 `sessionRef + rawPointers`：

- `sessionRef` 标识“这条会话来自哪个来源、哪种存储、主定位信息是什么”。
- `rawPointers` 标识“规范化消息/事件对应原始文件行、JSON 路径或数据库行”。

第一阶段只做来源注册、目录发现、路径校验和 Claude/Codex 无行为迁移，不改变现有历史行为。新来源可以先以 shadow/research parser 形式验证，但只要标记为正式已解析来源，就必须进入转换体系：可作为 `convertFrom`，并登记对应原生 writer 路线图。架构不能把新来源永久设计成只读，后续必须能兼容修改、删除、转换。

Source id 统一短 id。显示名不写死在业务逻辑里，由 descriptor + i18n 决定。

## 术语与边界

| 术语 | 定义 |
|---|---|
| CLI 工具 | 用户在项目终端中启动的命令行工具；不等于历史来源 |
| 历史来源类型 | 使用稳定短 id 标识的一类外部历史生产者，例如 `claude`、`codex` |
| 来源实例 | 某来源类型在特定运行环境和存储位置上的具体实例 |
| 逻辑会话 | 由来源原生会话身份界定、可能跨多个物理制品的一次对话单元 |
| 会话制品 | 组成逻辑会话的文件、数据库 row、registry 记录、附件或辅助状态 |
| 会话关系 | 有顺序、可多层的 parent/subagent、fork 或 continuation 关系 |
| Session Ref | 来源类型、来源实例、来源会话 id 和存储定位组成的稳定身份 |
| Raw Pointer | 规范化实体在文件行、JSON path 或数据库 row 中的原始定位 |
| 统一中间模型 | parser 输出、writer 消费的来源无关会话结构 |
| 目标会话包 | 目标工具识别、展示和 resume 一条会话所需的全部文件、registry 和数据库制品 |
| History Index DB | 可从外部历史重建的本地派生 SQLite 索引，不是真实来源 |
| Mutation Plan | 外部修改前描述影响、冲突、损失、备份和恢复方式的预览 |
| Mutation 备份 | 外部修改前保存在来源运行环境中的恢复快照，不是索引库备份 |
| 转换丢弃 | 经确认后不写入目标但保持来源不变的 subagent 或其他内容 |
| `completedWithOmissions` | 主会话成功但存在转换丢弃的结果状态 |

## Recall 调研结论

调研对象：`https://github.com/samzong/Recall`，当前 HEAD 为 `de6db943b05e61d92648e024150a1731403d4e63`。

Recall 的 adapter 模型值得参考，但不能照搬为 CLI-Manager 的完整能力模型：

- `SourceAdapter` 提供 `id()`、`label()`、`scan()`、`scan_for_sync()`、`prune()`、`resume_command()`、`app_command()`。
- `RawSession` 有 `source_id`、`directory`、`started_at`、`updated_at`、`entrypoint`、`messages`、`usage_events`、`events`、`source_file_path`、`custom_title`、`summary`、`duration_minutes`。
- `source_file_path` 是索引级溯源字段，用于同步、排除规则和导出；不是通用原始定位模型。
- Recall 的 `delete_session_data` 删除的是 Recall 自己 SQLite 索引中的 `sessions/messages/usage/events`，不是删除外部 Claude/Codex/Gemini 原始历史。
- Recall 的 `update_session_fields` 只更新索引库元数据，例如标题、摘要、时长、`source_file_path`。
- Recall 的 import/export 是 Recall JSONL 记录迁移；导入时 `is_import=true`，导入会跳过已有 `(source, source_id)`，不会合并或写回外部历史。
- TUI 单会话 export 是渲染纯文本，不是跨来源源格式转换。
- Recall 对其已知版本的 OpenCode/Kiro 使用只读 SQLite；本机 Kiro 当前版本实际是 workspace-sessions JSON，因此只能参考 adapter 思路，不能照搬其 Kiro 存储假设。
- `prune()` 用于清理 Recall 索引中的派生/过期记录，例如 Grok 子会话；不是外部源 mutation API。
- imported session 会禁止 resume/app open，但仍可 handoff。

结论：Recall 适合作为“多来源扫描/同步/索引”的参考；CLI-Manager 要支持修改、删除、转换，必须额外设计 adapter 级 mutation API、原始定位、备份和冲突检测。

## 当前问题

当前历史功能有几个硬绑定：

- 后端 `HistoryRoots` 只包含 `claude_config_dir` 和 `codex_config_dir`。
- 前端历史 IPC 参数只传 `claudeConfigDir` 和 `codexConfigDir`。
- `HistorySource` 类型只允许 `claude | codex`。
- 历史目录复用了 Hook 配置目录。
- `history_source_base`、会话校验、删除、转换都只接受 Claude/Codex。
- 新建项目的 CLI 工具选项只内置 `claude` 和 `codex`。
- 规范化会话依赖单个 `file_path`，无法表达数据库、混合存储、文件行、JSON path、数据库 row。

这些绑定会导致一个问题：新增来源如果只加扫描器，其他来源会被误当作 Codex/Claude 处理，尤其是项目匹配、resume、删除、转换、实时统计这些链路。

## 来源目录模型

每个历史来源必须由用户在 `设置 -> 历史来源` 中确认一个当前读取位置。同一来源不能同时启用多个 Windows、WSL 或自定义位置。自动探测只提供候选，不直接加入扫描范围。

建议模型：

```ts
type HistoryLocationPurpose = "config" | "content" | "state";
type HistoryLocationKind = "directory" | "database";

type HistoryLocationSlot = {
  id: string;
  labelKey: string;
  defaultLabel: string;
  purpose: HistoryLocationPurpose;
  kind: HistoryLocationKind;
  required: boolean;
};

type HistorySourceEnvironment =
  | { kind: "windows" }
  | { kind: "wsl"; distro: string }
  | { kind: "macos" }
  | { kind: "linux" };

type HistorySourceInstanceSettings = {
  id: string;
  environment: HistorySourceEnvironment;
  locations: Record<string, string>;
};

type HistorySourceSettings = {
  enabled: boolean;
  activeInstance?: HistorySourceInstanceSettings;
};
```

说明：

- `config`：工具配置根目录，例如 `.claude`、`.codex`、`.gemini`。
- `content`：直接存会话文件的目录，例如 Copilot `session-state`、Grok `sessions`。
- `state`：辅助状态库或索引，例如 Cursor `state.vscdb`、OpenCode `opencode.db`。
- `directory` 和 `database` 分开，避免后续 SQLite 来源硬塞进目录扫描逻辑。
- 用户只配置根路径，不配置派生文件，例如 `history.jsonl`、`summary.json`、`archived_sessions`。
- 一个来源类型同一时间最多一个 active instance。用户可以在 Windows、某个 WSL distro 或自定义位置之间切换，但不能同时读取多个实例。
- `activeInstance.id` 在当前配置期间保持稳定；切换位置后生成新的 instance id，防止新旧路径索引混淆。WSL 环境标识必须包含 distro。
- 一个 active instance 可以组合多个不同槽位，例如 Cursor 的 `projectsRoot + stateDb + workspaceStorageRoot`；每个槽位只允许一个位置。
- WSL UNC 路径仍允许直接输入；设置层从 `\\wsl$\<distro>\...` 或 `\\wsl.localhost\<distro>\...` 推断环境和 distro，不要求用户额外选择运行环境。
- Claude/Codex 升级迁移时把当前实际使用目录写入 active instance，避免升级后历史突然为空；新增来源必须先由用户确认读取位置才能启用。
- 更换读取位置采用两阶段切换：新位置验证并完成首轮索引后才替换 active instance；失败或取消继续使用旧位置，不让历史列表突然变空。

## sessionRef 与 rawPointers

`file_path` 只能表达文件型来源，不能表达 SQLite、混合来源、单条消息原始位置。需要升级为：

```ts
type HistoryStorageKind = "file" | "database" | "mixed";

type HistorySessionRef = {
  sourceId: HistorySourceId;
  sourceInstanceId: string;
  sessionId: string;
  storageKind: HistoryStorageKind;
  primaryPath?: string;
  databasePath?: string;
  rawKey?: string;
  sourceVersion?: string;
};

type HistoryRawPointer = {
  kind: "file" | "fileLine" | "jsonPath" | "databaseRow";
  path?: string;
  line?: number;
  jsonPath?: string;
  databasePath?: string;
  table?: string;
  rowId?: string;
  column?: string;
};
```

规则：

- `sourceId` 只表示工具类型；`sourceInstanceId` 表示当前已配置读取位置。稳定身份是 `(sourceId, sourceInstanceId, sessionId)`，避免用户切换 Windows/WSL/自定义位置后新旧索引混淆。
- 会话级别保存 `sessionRef`。
- 消息、usage event、tool event 可选保存 `rawPointers`。
- UI 的“打开原始位置”“查看源数据”“生成修改预览”必须基于 `rawPointers`，不要反推路径。
- 对没有稳定 raw pointer 的来源，不开放原地编辑和物理删除；只要统一中间模型达到最低完整度，仍可作为转换 `from`。作为 `to` 必须实现该来源的原生 writer。

## 本地索引库策略

数据库详细 schema、增量同步、迁移和查询方案见 [`docs/历史索引库设计.md`](./历史索引库设计.md)。

当前 CLI-Manager 已经不是完全每次全量读取所有历史文件：

- 后端有内存 + JSON 持久化的旧索引：`history-index-cache`，按文件 fingerprint 复用解析结果。
- 后端已有 SQLite catalog：`history-catalog.db`，表包括 `history_catalog_sessions`、`history_catalog_messages`、FTS 表和 `history_catalog_state`。
- `history_list_sessions` 和 `history_search` 优先走 SQLite catalog，失败时回退 legacy 文件索引。
- `history_get_session`、编辑、转换等详情链路仍会回读原始文件并重新构建详情。
- 现有 catalog schema 仍以 `file_path` 为核心主键，只适合 Claude/Codex JSONL 文件来源，不适合数据库/混合来源。

多来源方案应该正式采用 Recall 类似的“本地 SQLite 索引库”思路，但不要照搬 Recall 的 schema：

```text
外部历史源
  -> adapter 扫描/解析
  -> 统一中间模型
  -> CLI-Manager History Index DB
  -> 列表 / 搜索 / 统计 / 转换候选 / raw pointer 定位
```

建议直接在现有 `history-catalog.db` 中增加版本化 v2 表，把它升级为正式 History Index DB，避免同时维护第二个 SQLite 文件。核心表按 `sessionRef` 设计，而不是按 `file_path` 设计：

```sql
history_source_instances(
  id PRIMARY KEY,
  source_id,
  environment_kind,
  environment_key,
  locations_json
)

history_sessions(
  id INTEGER PRIMARY KEY,
  source_instance_id,
  source_session_id,
  storage_kind,
  primary_path,
  database_path,
  raw_key,
  project_key,
  cwd,
  title,
  created_at,
  updated_at,
  message_count,
  parser_version,
  fingerprint_value,
  model_version,
  last_seen_generation,
  indexed_at,
  UNIQUE(source_instance_id, source_session_id)
)

history_messages(
  id INTEGER PRIMARY KEY,
  session_id,
  message_index,
  role,
  timestamp,
  display_content,
  raw_pointers_json,
  UNIQUE(session_id, message_index)
)

history_message_parts(...)
history_tool_events(...)
history_usage_events(...)
history_file_changes(...)
history_source_state(...)
history_sync_runs(...)
```

原则：

- 列表、搜索、统计优先读本地索引库，不直接扫源目录。
- 刷新索引时只扫描目录结构和 fingerprint，只有新增/变更的 session 才重新解析。
- 页面请求不触发全量解析。刷新由应用启动后台同步、显式刷新、来源设置变化以及后续文件监听/轮询触发。
- adapter 的发现阶段只返回轻量 `sessionRef + fingerprint`；文件来源优先用 path/mtime/size，必要时补 hash，数据库来源使用稳定 row key、更新时间或数据库变更标记。
- fingerprint 变化或 `parser_version` 升级时，在单个短事务内替换该会话的 session/messages/events/usage/FTS。
- 只有来源扫描成功且确认覆盖完整范围时，才删除“上次存在、本次缺失”的索引记录。路径离线、WSL 未启动、权限失败时只标记来源不可用，不能误删索引。
- 详情页可以优先从索引库读规范化内容；需要展示完整工具调用、diff、raw view 或执行 mutation 时，再按 `rawPointers` 回读原始源。
- 编辑、删除、转换不直接改索引库。它们先改外部源，成功后重新解析并更新索引库。
- 外部写入成功但索引刷新失败时，标记为“外部已完成、索引待刷新”并重试，不能通过回写旧索引掩盖真实状态。
- 索引库是派生数据，可以删除重建；外部历史源仍是真实来源。
- 数据库/混合来源也进入同一个索引库，避免 UI 为每类来源写独立查询逻辑。
- SQLite 使用 WAL、busy timeout 和短事务；源解析在事务外完成，避免大历史库刷新时长期阻塞查询。
- v2 稳定后淘汰 `history-index-cache` 运行时路径，避免 SQLite 与 JSON 双索引长期产生一致性问题。

## 来源写入复杂度复审

Recall 只能证明这些来源“当前可读”，不能证明可以安全写回。每个 `convertTo/edit/delete` adapter 都需要独立验证实际安装版本、artifact graph、共享状态和目标工具可发现性。

| 来源 | 逻辑会话主要制品 | writer 额外关注 | 风险 |
|---|---|---|---|
| Claude | project/transcript JSONL、`sessions-index.json`、subagents | parent/child 关系、索引摘要、resume 可发现性 | 中 |
| Codex | rollout JSONL、`history.jsonl`、`session_index.jsonl`、`state_5.sqlite` | 文件 + 两个共享 registry + DB row 的跨资源一致性 | 高 |
| Gemini | `tmp/**/chats/*.json` | chat schema/version、项目 hash/目录归属 | 中 |
| Copilot | session 目录与 `events.jsonl` | 必须生成合法事件序列，不能只写展示消息 | 中 |
| Antigravity | conversation transcript、共享 `history.jsonl` workspace 映射 | conversation id 与 workspace registry 同步 | 高 |
| Grok | `updates.jsonl`、`summary.json`、可能的 subagent 会话 | summary 与 updates 一致、父子会话范围 | 中 |
| Pi | 版本化 session JSONL、`settings.json` 指向的自定义 session dir | 按文件内 version 写入，不能假定固定默认目录 | 中 |
| OpenCode | `session/message/part` 等 SQLite row graph | 外键/JSON part/schema version、活动 DB 锁 | 高 |
| Kiro | `conversations_v2` 的 key/value row | blob schema、应用版本、恢复与 resume 语义尚不稳定 | 高 |
| Cursor | agent transcript、global/workspace `state.vscdb`、composer/bubble rows | 多 DB、多 workspace、共享 KV、版本漂移 | 极高 |
| Cline | task 目录、`ui_messages.json`、`task_metadata.json` | task id/时间语义、插件状态与 metadata 一致 | 中 |

结论：

- parser 完成不代表 writer 可直接照着反序列化结构实现。
- 每个 writer 交付前必须用目标工具真实版本创建基准会话，确认最小必需制品和 schema。
- 文件型来源也不天然低风险；存在共享 registry、父子会话或版本化事件协议时仍需 mutation manifest。
- OpenCode、Cursor 等新增数据库/混合 writer 不能靠通用“实验性”开关放行；必须完成 source-specific writer 验证、备份恢复能力、schema/version 校验和目标运行时锁检测后，才能把 `convertTo/edit/delete` 标记为 `supported`。Kiro 当前 JSON 格式按文件型来源处理。Codex 保留现有已支持转换能力，但后续新增 edit/delete 或扩展 state DB 写入时仍必须经过 plan、备份和版本校验。

## 能力模型

不要用“是否 Claude/Codex”判断操作。使用 descriptor 声明能力。

```ts
type HistoryCapabilityState = "supported" | "planned" | "unsupported";

type HistorySourceCapabilities = {
  list: HistoryCapabilityState;
  search: HistoryCapabilityState;
  stats: HistoryCapabilityState;
  usage: HistoryCapabilityState;
  rawOpen: HistoryCapabilityState;
  resume: HistoryCapabilityState;
  appOpen: HistoryCapabilityState;
  edit: HistoryCapabilityState;
  delete: HistoryCapabilityState;
  convertFrom: HistoryCapabilityState;
  convertTo: HistoryCapabilityState;
  realtimeStats: HistoryCapabilityState;
};
```

来源 capability 不是最终操作权限。UI 和后端都要计算：

```text
effective capability = source capability ∩ session-level constraints
```

例如来源支持 `edit/delete/convertFrom`，但 relation kind 为 `subagent` 的会话仍禁止单独编辑、删除和转换。

第一版策略：

- Claude/Codex 保持现有能力。
- 新来源先以 shadow/research parser 验证；正式标记为已解析来源后，至少开放已验证的 `list/search/stats/rawOpen`，并进入 `convertFrom` 候选和 writer 路线图。
- `edit/delete/convertFrom/convertTo` 可以是 `planned`，但不能在类型或存储结构上被设计死。
- 每个已接入 parser 的来源都必须登记原生 writer 的后续任务，最终支持作为 `to`；不能用 capability 把数据库或混合来源永久排除。
- UI 根据 capability 决定是否显示入口；后端再次校验，不能只靠前端隐藏。

## 修改、删除、转换的长期设计

备份创建、保留清理、自动回滚、人工恢复和设置页入口拆分到独立子任务 `07-14-history-backup-restore`，详细设计见 [`docs/历史备份与恢复设计.md`](./历史备份与恢复设计.md)。本任务只定义 mutation 必须依赖的 backup/restore contract。

用户要求不能把新来源永久设计成只读，因此需要预留 mutation 层。

建议后端增加 adapter 级 mutation trait：

```rust
trait HistorySourceMutationAdapter {
    fn build_delete_plan(&self, session_ref: &HistorySessionRef) -> Result<HistoryMutationPlan>;
    fn build_edit_plan(&self, pointer: &HistoryRawPointer, patch: HistoryEditPatch) -> Result<HistoryMutationPlan>;
    fn build_convert_plan(&self, session_ref: &HistorySessionRef, target: HistorySourceId) -> Result<HistoryMutationPlan>;
    fn build_operations(&self, plan: &HistoryMutationPlan) -> Result<Vec<HistoryMutationOperation>>;
    fn build_restore_operations(&self, manifest: &HistoryBackupManifest) -> Result<Vec<HistoryMutationOperation>>;
}
```

要求：

- 所有 mutation 必须先生成 plan，再执行。
- plan 必须包含目标文件/数据库、影响范围、备份路径、冲突校验条件。
- adapter 只负责生成 primitive operations、preflight 校验和逆操作；中央 `MutationExecutor` 统一状态机、备份、提交、回滚、恢复和定向重新解析。
- 文件型来源用 mtime/size/hash 防止覆盖用户外部修改。
- 数据库来源用事务执行，必要时要求用户关闭源应用或切只读。
- 混合来源由 adapter 自己决定文件和数据库的修改顺序。
- 删除默认语义是物理删除：文件型来源删除/重写原始记录，数据库来源删除 row 或执行来源要求的等价物理删除。
- 物理删除主会话时级联包含全部 subagent 后代。plan 必须展示会话树、制品和数量，并在任何删除开始前完成整棵树的 preflight 与备份。
- 级联删除采用全有或全无语义：任一 subagent/制品失败就恢复已删除部分，整次 mutation 失败，不允许部分删除成功。
- subagent 是只读关联节点，不允许单独编辑或物理删除。级联删除只能以主会话为 mutation root，由后端内部包含 subagent，不能绕过限制直接提交 child session ref。
- 物理删除执行前必须备份。默认备份目录是“历史来源所在运行环境”的用户主目录下 `.cli-manager/backups`，不能只按 Windows 路径处理。
- 转换不做 Claude/Codex 特例。凡是已经解析进统一中间模型的来源，都可以作为 `from`；凡是 adapter 支持写入目标原生格式的来源，都可以作为 `to`。UI 允许用户选择任意可行 `from -> to` 组合，实际可选项由来源 capability 决定。
- 转换固定是非破坏性复制，成功或失败都不修改来源。禁止“转换后自动删除来源”；需要删除时必须另起物理删除 plan、备份和确认。
- 转换始终生成新的目标原生会话 id，不覆盖、不合并已有会话。该 id 必须贯穿目标会话包的全部制品；重试通过 idempotency key 复用同一 id。
- 转换必须生成兼容性报告，列出目标格式无法表达的 tool call、usage、diff、附件或来源私有字段。无损时可直接执行；有损时提醒用户并要求明确确认后才能继续。每个 parser 可在统一模型中保留命名空间化 source extension，避免过早丢失来源特有数据。

默认备份目录解析：

| 环境 | 默认目录 |
|---|---|
| Windows 原生来源 | `%USERPROFILE%\.cli-manager\backups`，例如 `C:\Users\<user>\.cli-manager\backups` |
| WSL 来源 | 对应 distro 内的 `$HOME/.cli-manager/backups`，例如 `/home/<user>/.cli-manager/backups` |
| macOS 来源 | `$HOME/.cli-manager/backups`，例如 `/Users/<user>/.cli-manager/backups` |
| Linux 来源 | `$HOME/.cli-manager/backups`，例如 `/home/<user>/.cli-manager/backups` |

规则：

- 备份目录跟随被修改的历史来源所在环境，而不是固定跟随 CLI-Manager UI 所在系统。
- 如果来源路径是 WSL UNC，例如 `\\wsl$\<distro>\home\<user>\...`，备份必须写入该 distro 的 Linux home。
- 如果来源路径是 Windows 本地路径，备份写入 Windows `%USERPROFILE%`。
- 备份记录必须保存原始 `sessionRef`、`rawPointers`、源文件/数据库快照、执行前 hash/mtime/size。

阶段性能力：

1. Phase M1：只做 plan/dry-run，不真正写外部源。
2. Phase M2：Claude/Codex 文件型删除和编辑。
3. Phase M3：Gemini/Copilot/Grok/Pi/Cline 文件型来源。
4. Phase M4：OpenCode/Cursor 数据库或混合来源，必须带事务、备份、锁冲突处理；Kiro 当前 JSON 格式走文件型 mutation。
5. Phase M5：跨来源转换矩阵。已解析来源默认进入 `from` 候选；`to` 候选按目标来源 `convertTo/write` capability 开放。不按 Claude/Codex 写死；最终为全部已解析来源补齐原生 writer。

## 数据库/混合来源写入闸门

不设置通用“实验性写入”开关。数据库/混合来源是否允许真实写入，只由 capability、格式版本、Backup Service 能力和目标运行时状态共同决定。

规则：

- `convertTo/edit/delete` 必须按 `sourceId + format/schema version + adapter version` 声明能力。
- 未验证 schema、缺少完整 artifact graph、缺少 restore strategy、目标工具运行中或 backup root 不可用时，直接阻止 mutation。
- plan/dry-run 可以开放给研究和预览，但不能创建任何目标 artifact、registry row 或数据库 row。
- 用户确认有损转换只能接受语义损失，不能绕过结构性不可写错误。
- 数据库/混合 writer 首次交付必须有真实目标工具版本的基准会话对照和恢复测试。

理由：写坏外部 SQLite 或混合状态后，单靠一个设置开关不能提供安全性。安全边界必须落在 writer 验证、备份恢复和运行时锁检测上。

页面能力：

可以内置更多选项，但不要直接等同于历史来源。

项目的 `cli_tool` 表示“启动什么命令”；历史来源表示“从哪里解析历史”。两者可以有关联，但不是同一个概念。

首批可内置：

## 风险

| 风险 | 处理 |
|---|---|
| 来源增多后旧缓存 key 失效 | 第一阶段显式升级 cache key，不复用旧结构 |
| 单个 `file_path` 无法覆盖 SQLite/混合来源 | 升级为 `sessionRef + rawPointers` |
| 项目来源筛选误判 | 用 `historySourceId`，不要靠字符串包含推断 |
| SQLite 被占用 | 第一版只读打开；mutation 阶段增加事务、备份和锁冲突提示 |
| mutation 写坏外部历史 | 先 dry-run plan，执行前备份，执行后重新解析校验 |
| 删除语义不一致 | 删除能力由 adapter 定义，不做通用文件删除 |
| 转换后目标工具不可 resume | 转换结果标注来源、原始 ref、兼容等级，不默认承诺原生 resume |
| WSL 目录扫描回退到 Windows API | WSL UNC 继续走现有 `wsl.exe` 扫描路径 |
| 一条会话跨文件、registry 和数据库 | adapter 枚举 artifact graph；mutation/备份/恢复按完整制品集合执行 |
| 转换多步写入中途失败 | mutation id、幂等 key、staging、manifest、逆操作和启动恢复 |
| 来源格式升级后 writer 写坏数据 | capability 按 source + format/schema version + adapter version 判定，未知版本默认禁写 |
| 自定义目录重叠或备份被重新扫描 | canonical locator 去重，排除备份/索引/mutation workspace，限制 symlink/junction |
| 统计口径跨来源不一致 | 保存 timestamp/usage/cost quality 和 pricing version，UI 区分 reported/estimated/unknown |
| 超大 tool payload 或附件撑爆索引 | message part 支持 inline/rawOnly/unavailable，转换时回源并校验 fingerprint |

## 方案复审补充

复审现有实现与 Recall adapter 后补充以下硬要求：

- “逻辑会话”和“物理制品”分离，新增 session artifacts 和 session relations。
- 保持现有历史列表的主会话/subagent 下挂效果，但由索引库显式 relation 替代 `file_path` 路径推断，并支持多层关系。
- artifact 标记 exclusive/shared；共享 registry 文件和数据库只修改目标 record/row，不能整文件删除或用旧快照覆盖其他会话的新数据。
- parser 和 writer 都必须绑定已验证的来源格式/schema 版本。
- 跨文件/数据库 mutation 必须幂等、可回滚、可在应用重启后恢复。
- 备份必须有 restore plan；只生成文件但无法恢复不算完成备份能力。
- 来源离线、路径重叠、symlink/junction、备份目录递归扫描需要显式处理。
- usage、cost、时间戳必须携带质量/来源口径，避免跨来源统计失真。
- 转换成功必须通过目标重解析验证，不能只以文件写入成功作为完成条件。
- writer 必须生成目标工具完整会话包，不能只写主历史文件或 CLI-Manager 自己的历史索引。
- 缺少必需制品、目标 schema 未知或版本未验证时阻止转换，且不能按普通有损项确认绕过；preflight 失败不得写入任何目标数据。
- 目标工具正在运行时阻止转换，不提供强制继续，也不主动结束目标进程；关闭后必须重新生成/校验 plan。
- `convertTo` 和 `resume` 是独立 capability。
- 主会话转换 plan 检查全部后代；目标能保留关系时按树写入，不能保留或转换失败的 subagent 作为有损丢弃项确认后不写入目标。禁止把 subagent 转成独立会话或压平进主会话，来源数据保持不变。
- subagent 使用部分成功语义：单个 subagent 失败只回滚并丢弃该子树，主会话和其他成功 subagent 继续；结果标记 `completedWithOmissions` 并列出实际丢弃项。
- subagent 不允许作为转换根。列表中单独选中 subagent 时禁用转换并提示从主会话发起，后端基于显式 relation 再次拒绝直接转换请求。
- subagent 不允许单独编辑或删除。列表、详情和批量操作中禁用对应入口；后端基于 relation 返回稳定错误，主会话级联删除除外。
- 物理删除主会话时级联删除完整 subagent 树，不能留下孤立 subagent。
- 级联删除中任一步失败都回滚整棵树；回滚失败进入人工恢复并锁定该来源实例的后续 mutation。

## 已拍板

- 外部历史 mutation 备份目录放在历史来源所在运行环境的用户主目录下 `.cli-manager/backups`：Windows 为 `%USERPROFILE%\.cli-manager\backups`，WSL/macOS/Linux 为 `$HOME/.cli-manager/backups`。
- 删除默认语义是物理删除。
- 物理删除主会话时关联 subagent 全部级联删除。
- 主会话树删除为全有或全无，任一子任务失败则恢复全部已删除内容。
- subagent 不允许单独编辑、删除或转换，只能查看；删除主会话时作为级联范围处理。
- 跨来源转换始终保留来源，不提供隐式迁移或转换后自动删除。
- 转换创建新的目标原生 id，不覆盖、不合并已有目标会话；同一 id 必须在目标完整会话包中一致。
- 目标会话包不完整或 schema 不受支持时不允许尽力写主文件，必须先修复/初始化目标环境。
- 目标工具运行期间不允许转换，任何 capability 或用户确认都不能绕过。
- 转换要覆盖所有已解析来源：已解析来源可作为 `from`，具备目标写入能力的来源可作为 `to`，用户可自定义选择任意可行组合。
- 数据库/混合来源写入不设置通用实验性开关；必须由 source-specific capability、schema/version 校验、备份恢复和运行时锁检测共同放行。
- 有损转换允许执行，但必须列出损失并由用户明确确认。
- History Index DB 不加密，使用普通本地 SQLite，不引入 SQLCipher。
- Mutation 备份按每个运行环境的 backup root 独立计算，默认保留 7 天或最多占用 1 GiB，任一条件先达到即清理；提供立即永久清理。
- 单次必要备份预计超过该环境 1 GiB 时阻止 mutation；用户必须更换备份目录或显式临时提高仅本次 mutation 的环境上限，禁止跳过备份或静默超限。


## 不做

- 不新增依赖。
- 不升级 Tauri、React、Rust 版本。
- 不照抄 Recall parser；每个来源按本项目统一模型逐批实现和验收。
- 不把 Hook 设置和历史目录继续绑在一起。
- 不把所有历史来源都放进新建项目 CLI 选项。
- 不在历史来源索引第一阶段实现备份管理和恢复 UI；由独立 Trellis 子任务负责。
