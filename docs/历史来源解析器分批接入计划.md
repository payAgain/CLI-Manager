# 历史来源解析器实施计划

本计划从 2026-07-17 起进入真实 parser 实施，不再把 descriptor、roadmap 或 shadow schema 记作“已完成解析”。

## 当前基线

- 真正可解析的来源已覆盖 Claude、Codex、Gemini、Kiro、OpenCode。
- `scan_session_files()` 覆盖文件型来源；OpenCode 通过 catalog 刷新只读解析 SQLite，并使用虚拟 locator 兼容旧列表/详情 API。
- v2 索引是 shadow path：它从旧 catalog 搬运 Claude/Codex 数据，不是通用 source adapter。
- `active_v2_source_instances()` 仍限定 `source_id IN ('claude', 'codex')`。
- Copilot、Antigravity、Grok Build、Pi、Cursor、Cline 仍只有注册、探测或 capability 元数据，没有真实 parser。

## 完成定义

每个来源必须完成以下整条链路，才允许把 `list/search/stats/usage/rawOpen` 对应能力改为 `supported`：

```text
discover -> fingerprint -> parse -> v2 index -> list/detail/search/stats -> golden fixture
```

仅完成路径探测、schema 调研、descriptor 或测试样本，不算 parser 完成。单个来源解析失败必须写入 `history_index_failures`，不能阻断其他来源刷新。

## 核心改造：先清掉双索引阻塞

1. 建立轻量 `HistorySourceAdapter` 注册表，只定义 `discover`、`fingerprint`、`parse`；mutation/writer 不混入 parser 接口。
2. v2 refresh 直接遍历 active source instances 和 adapter，不再从 legacy catalog 二次搬运。
3. 为 v2 补齐列表、详情、搜索、统计查询；新来源不允许接回 `file_path` 专用旧链路。
4. Claude/Codex 先通过 adapter 跑 parity，确认同 fixture 的会话数、消息数、token 和项目归属一致后切换 v2 read path。
5. 读取位置统一来自 `history_source_instances.locations_json`；移除新增来源对 `claude_config_dir/codex_config_dir` 参数的依赖。
6. 保留旧 catalog 作为短期回退，所有来源 parity 通过后再单独删除，避免在 parser 任务里顺手大重构。

## 实施批次

| 批次 | 来源 | 当前样本 | 本批目标 |
| --- | --- | --- | --- |
| core | Claude、Codex | 已有 fixtures | adapter 化、v2 主读链路、旧行为 parity |
| batch-1 | Gemini、Kiro | 本机有真实 JSON | 首批新增 parser；优先解决用户实际 Kiro 历史不可见 |
| batch-2 | OpenCode、Antigravity | 本机有 DB/目录 | SQLite snapshot-safe 读取与 Antigravity 格式确认 |
| batch-3 | Copilot、Grok Build、Pi、Cline | 待补稳定样本 | 每个来源先锁定版本化 fixture，再实现文件型 parser |
| batch-4 | Cursor | 待补混合存储样本 | workspace/state DB 多制品关联与 raw pointers |

批次表示合并顺序，不表示后续来源可以缺席。所有来源从现在起都进入 active implementation task；缺样本的来源状态是“fixture blocked”，不是“规划完成”。

## 来源级实现要点

### Gemini

- 默认发现：`~/.gemini/tmp/*/chats/session-*.json`。
- 已确认顶层字段：`sessionId/projectHash/startTime/lastUpdated/messages`。
- `messages[]` 至少映射 `id/timestamp/type/content`，项目路径优先从 tmp project metadata 解析，无法确认时保留 project hash，禁止猜路径。

### Kiro

- Windows 默认发现：`%APPDATA%/Kiro/User/globalStorage/kiro.kiroagent/workspace-sessions`。
- 每个 workspace 子目录包含 `sessions.json` 和 `<sessionId>.json`；`sessions.json` 只作 registry，不作为会话正文。
- 已确认会话字段：`sessionId/title/workspaceDirectory/workspacePath/selectedModel/history`。
- `history[].message` 映射角色与内容；内容同时兼容 string 和 `{type,text}[]`。
- 当前安装格式是 JSON，不是原规划中的 `data.sqlite3`。parser 和 fixture 必须按 JSON 版本实现。

### OpenCode

- 默认发现：`~/.local/share/opencode/opencode.db`。
- 已落地只读 SQLite parser，实际表为 `session/message/part`。
- catalog 使用 `<opencode.db>#session=<session_id>` 虚拟 locator，避免一个 DB 内多会话互相覆盖。
- `message.data.tokens` 映射 input/output/reasoning/cache.read/cache.write；reasoning 计入 output。
- edit/delete/convert/resume 保持禁用，后续切 v2 read path 时再补多 artifact raw pointers。

### Antigravity

- Windows 候选根：`%APPDATA%/Antigravity`。
- 先枚举会话制品与版本标记，确定文件/数据库/混合类型后冻结 fixture，再写 adapter。

### Copilot

- 默认历史：`~/.copilot/session-state/<session-dir>/events.jsonl`。
- 事件类型：`session.start`、`user.message`、`assistant.message`、`tool.execution_start`、`tool.execution_complete`。
- session id 优先 `session.start.data.sessionId`，cwd 来自 `session.start.data.context.cwd`；缺少 start 事件时回退目录名。

### Grok Build、Pi、Cline

- 每个来源单独收集至少一个正常会话和一个 tool-call 会话。
- 样本缺失时可以完成 discover 骨架和稳定错误码，但不得把 capability 标成 supported。
- Grok 来源产品名统一为 `Grok Build`，source id 保持 `grok`。

### Cursor

- 按 workspace storage + state DB 作为混合来源处理。
- 已确认涉及 `%APPDATA%/Cursor/User/globalStorage/state.vscdb`、`%APPDATA%/Cursor/User/workspaceStorage/*/state.vscdb` 与 `~/.cursor/projects/*/agent-transcripts/<sessionId>/<sessionId>.jsonl`。
- SQLite 键包括 `cursorDiskKV` 的 `composerData:<composerId>`、`bubbleId:<composerId>:<bubbleId>`；主身份使用 `composerId`。
- 一个逻辑会话可以关联多个 artifact；必须输出 database row/file raw pointers，不能伪造成单个 `file_path`。

## 每个子任务的测试矩阵

- discover：默认路径、自定义路径、不存在、离线、空目录。
- parse：正常消息、tool call、usage、无时间戳、损坏记录、超大内容。
- identity：同 session id 在不同 source instance 不冲突。
- incremental：未变跳过、新增写入、修改重解析、删除失效、parser version 重建。
- query：全部/单来源列表、详情、全文搜索、项目过滤、统计聚合。
- safety：只读 parser 不修改原始来源；fixture 必须脱敏，测试不得依赖用户实时目录。

## 验收与晋级

来源只有同时满足以下条件才算完成：

- golden fixture 和 parser 单测通过。
- v2 session/message/tool/usage/raw pointer 写入正确。
- 列表、详情、搜索、统计至少各有一条集成测试。
- UI 来源下拉框能筛出该来源，空结果与错误状态正确。
- capability 从 `planned` 改为 `supported` 的 diff 与上述测试在同一任务交付。
- 对应 writer 仍可保持 `planned`；parser 完成不等于允许 edit/delete/convertTo。

## 明确不在 parser 批次中做

- 不新增依赖；复用 `serde_json`、`sqlx` 和现有 SQLite。
- 不同时实现外部 mutation 或原生 writer。
- 不因顺手而删除 legacy catalog；切换和清理由独立验收项控制。
- 不使用用户真实会话作为持续集成测试输入。
