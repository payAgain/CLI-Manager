# 历史备份与恢复设计

## 目标

为历史会话的编辑、物理删除和目标侧转换写入提供独立的 Backup Service，覆盖：

- 跨 Windows、WSL、macOS、Linux 的备份根解析。
- 文件、SQLite、混合 artifact graph 的一致性快照。
- mutation 自动回滚。
- 用户手动恢复。
- 7 天 / 1 GiB 保留清理。
- 设置页中的备份查看、恢复和永久清理。

该模块不负责解析历史格式，也不负责生成来源 writer；它消费 mutation plan 中的 artifact graph、fingerprint、backup/restore operation。

## 已确认规则

- 备份是修改外部原始历史前的恢复快照，不是 History Index DB 备份。
- 默认备份根位于被修改来源所在运行环境的用户主目录：
  - Windows：`%USERPROFILE%\.cli-manager\backups`
  - WSL：对应 distro 的 `$HOME/.cli-manager/backups`
  - macOS/Linux：`$HOME/.cli-manager/backups`
- 每个运行环境的 backup root 独立计算配额。
- 默认保留 7 天或最多占用 1 GiB，任一条件先达到即清理。
- 单次必要备份预计超过 1 GiB 时阻止 mutation；用户只能更换该环境备份目录，或显式提高仅本次 mutation 的临时上限。
- 临时超额备份不做长期豁免；mutation 完成后继续按默认 7 天 / 1 GiB 自动清理。
- 备份保持来源原始格式，不加密、不参与同步。
- 目标工具运行时禁止转换或恢复，不允许 override，也不由 CLI-Manager 强制结束进程。
- 物理删除主会话时备份覆盖主会话及全部 subagent 后代。
- 主会话树删除全有或全无；任一步失败恢复整棵树。
- 转换是非破坏性复制，不备份来源；只备份会被修改的目标 shared artifact。

## 模块边界

```text
HistorySourceMutationAdapter
  -> HistoryMutationPlan + artifact graph
  -> BackupService.preflight()
  -> BackupService.create_backup()
  -> MutationExecutor.apply()
  -> verify
       success -> BackupService.mark_completed()
       failure -> BackupService.build_restore_plan()/restore()
```

依赖：

- 来源 adapter 提供完整 artifact graph、ownership、fingerprint 和版本信息。
- mutation executor 提供 mutation id、idempotency key、操作顺序和逆操作。
- 环境解析层提供 Windows/WSL/macOS/Linux home 和目标进程检测。
- History Index DB 在恢复完成后执行定向重新解析。

不做：

- 不自行猜测来源文件或数据库表。
- 不通过备份库替代外部历史真实来源。
- 不实现跨机器或云端备份。
- 不承诺 SSD/文件系统层面的法证级不可恢复。

## 设置模型

```ts
type HistoryBackupEnvironmentKey =
  | "windows"
  | `wsl:${string}`
  | "macos"
  | "linux";

type HistoryBackupEnvironmentSettings = {
  customRoot?: string;
};

type HistoryBackupSettings = {
  retentionDays: 7;
  maxBytes: 1073741824;
  environmentOverrides: Record<HistoryBackupEnvironmentKey, HistoryBackupEnvironmentSettings>;
};
```

规则：

- 默认值固定为 7 天和 1 GiB；未来是否允许永久修改配额另行设计。
- `customRoot` 只替换对应运行环境，不影响其他环境。
- 临时上限不写入 settings，只保存在当前 mutation plan 中。
- WSL override 必须是对应 distro 内的 Linux 路径；不能把 WSL mutation 备份到另一个 distro。
- 自定义备份根不能位于历史读取目录内部，避免被 parser 重新索引。

## 目录布局

```text
<backup-root>/
  <source-id>/
    <source-instance-id>/
      <yyyyMMdd-HHmmss>-<mutation-id>/
        manifest.json
        files/
        database/
        exports/
        workspace/
```

- 一个 mutation 对应一个完整目录，清理和永久删除以该目录为单位。
- `workspace/` 存放 staging 和恢复临时文件，不计为可独立清理项。
- manifest 更新使用临时文件 + atomic rename。
- manifest 不完整、执行中、恢复中的目录禁止自动清理。

## Manifest

```ts
type HistoryBackupManifest = {
  schemaVersion: 1;
  mutationId: string;
  idempotencyKey: string;
  operation: "edit" | "delete" | "convertTarget" | "restore";
  status:
    | "creating"
    | "ready"
    | "mutationApplying"
    | "completed"
    | "rollingBack"
    | "rolledBack"
    | "restoring"
    | "restored"
    | "manualRecoveryRequired";
  environmentKey: HistoryBackupEnvironmentKey;
  sourceId: string;
  sourceInstanceId: string;
  sessionRefs: HistorySessionRef[];
  adapterVersion: string;
  sourceFormatVersion?: string;
  createdAt: number;
  completedAt?: number;
  totalBytes: number;
  artifacts: HistoryBackupArtifact[];
};

type HistoryBackupArtifact = {
  artifactId: string;
  role: string;
  kind: "file" | "database" | "databaseRows" | "directory";
  ownership: "exclusive" | "shared";
  originalLocator: HistoryRawPointer;
  snapshotRelativePath?: string;
  beforeFingerprint: string;
  afterFingerprint?: string;
  restoreStrategy: "replace" | "reverseOperation" | "sqliteBackup" | "rowGraph";
};
```

Manifest 必须足够支持：

- 判断备份是否完整。
- 判断能否自动恢复。
- 列出受影响会话和 artifact。
- 防止恢复覆盖 mutation 后的新数据。
- 在应用重启后继续回滚或提示人工恢复。

## Backup Service 接口

```rust
trait HistoryBackupService {
    fn preflight(&self, plan: &HistoryMutationPlan) -> Result<HistoryBackupEstimate>;
    fn create_backup(&self, plan: &HistoryMutationPlan) -> Result<HistoryBackupManifest>;
    fn mark_mutation_completed(&self, mutation_id: &str) -> Result<()>;
    fn build_restore_plan(&self, mutation_id: &str) -> Result<HistoryRestorePlan>;
    fn restore(&self, plan: &HistoryRestorePlan) -> Result<HistoryRestoreResult>;
    fn list_backups(&self, environment: &str) -> Result<Vec<HistoryBackupSummary>>;
    fn cleanup(&self, environment: &str) -> Result<HistoryBackupCleanupResult>;
    fn delete_permanently(&self, mutation_id: &str) -> Result<()>;
}
```

所有 mutation executor 只能通过该接口创建备份，不能各 adapter 自己拼接备份路径和清理规则。

## Preflight

执行前检查：

- backup root 可解析、可创建、可写。
- 来源运行环境可用；WSL distro 必须在线。
- 预计快照大小、root 当前占用和磁盘剩余空间。
- 单次快照是否超过默认/临时上限。
- artifact fingerprint、权限和 ownership。
- 目标工具或被写应用是否已经关闭。
- restore strategy 是否存在；没有恢复方案则不能执行破坏性 mutation。

preflight 失败不创建任何外部目标 artifact，也不修改来源。

## 快照策略

### Exclusive 文件

- 复制完整文件或目录。
- 保存路径、mtime、size、hash 和权限信息。
- 恢复时目标未发生额外变化可 atomic replace。

### Shared 文件

- 已验证 append-only JSONL 可使用逻辑备份：保存截断 offset、尾部 hash、受影响 record 和 reverse operation。
- 未验证格式、破坏性重写或无法证明 record 边界时保存完整文件快照用于灾难恢复。
- 正常恢复优先执行逆操作，不能用旧整文件覆盖 mutation 后其他会话的新记录。

### SQLite

- 活动 WAL 数据库使用 SQLite online backup 或 adapter 验证过的一致性快照。
- 不能只复制主 `.db` 文件。
- 已验证 schema 允许 row graph 逻辑备份：保存受影响 row、外键边界、索引/registry 关联和 reverse operation。
- 未知 schema、跨版本未验证、缺少外键边界或破坏性迁移时，必须使用完整一致性快照。
- 正常恢复优先事务化恢复目标 row graph；整库替换只在 fingerprint 证明没有后续无关写入时允许。

### Mixed

- 同一个 mutation manifest 同时覆盖文件、数据库和 registry。
- 所有快照成功后 manifest 才能进入 `ready`。
- 任一快照失败删除本次未完成 workspace，不开始 mutation。

## 自动回滚

- mutation 任一步失败后进入 `rollingBack`。
- 按 mutation plan 的逆序恢复 shared/exclusive artifact。
- 每个恢复步骤重新检查 fingerprint。
- 恢复后通过来源 adapter 重新解析并验证会话树、消息数、registry 和数据库 row。
- 成功状态为 `rolledBack` 或业务结果 `failedRolledBack`。
- 失败状态为 `manualRecoveryRequired`，锁定该来源实例后续 mutation，但保留读取、备份导出和人工恢复。

主会话级联删除是全有或全无；任何 subagent 删除失败都恢复主会话和全部已删除子树。

## 手动恢复

入口：`设置 -> 历史来源 -> 备份与恢复`。

用户流程：

1. 选择运行环境。
2. 查看占用、备份时间、来源、操作类型、会话标题和状态。
3. 选择一条备份并打开恢复预览。
4. 检查目标工具已关闭、当前 fingerprint、将覆盖/逆向修改的 artifact。
5. 用户确认后执行恢复。
6. 重新解析来源并展示恢复结果。

限制：

- 目标工具运行时禁止恢复，无 override。
- 当前 artifact 与 manifest 的允许 fingerprint 不一致时阻止自动恢复，进入冲突详情；不能强制覆盖无关新数据。
- `manualRecoveryRequired` 提供导出 manifest、打开备份目录和再次生成恢复 plan。
- 恢复本身也使用新的 operation id 记录，不修改原 manifest 历史。

## UI

设置页按运行环境展示：

- 当前 backup root 和更换目录。
- 当前占用 / 1 GiB。
- 7 天保留说明。
- 最近备份列表和状态筛选。
- 恢复、打开位置、永久删除。
- 清理失败、环境离线、manifest 异常提示。

永久删除必须二次确认；删除执行中、恢复中或 `manualRecoveryRequired` 的备份默认禁止。

所有用户文案同步 `zh-CN` 和 `en-US`。

## 自动清理

清理触发：

- 应用启动后的低优先级维护任务。
- mutation 完成后。
- 用户手动清理。

规则：

1. 每个运行环境独立执行。
2. 先删除超过 7 天的 completed/rolledBack/restored 备份。
3. 再从旧到新删除，直到该 root 占用不超过 1 GiB。
4. 以完整 mutation 目录为单位删除。
5. creating、mutationApplying、rollingBack、restoring、manifest 异常目录不自动删除。
6. 环境离线时跳过该环境，不影响其他环境。
7. 清理失败不阻断历史读取，但设置页必须显示错误。
8. 通过临时上限完成的超额备份进入同一套自动清理；如果旧备份删完仍超过 1 GiB，本次 completed 备份也允许被自动删除。mutation 确认页和完成页必须提示可恢复窗口可能很短。

## 阶段

### B1：基础服务

- backup root resolver。
- versioned manifest。
- preflight/estimate。
- 文件型 snapshot。

### B2：保留与自动回滚

- 7 天 / 1 GiB 清理。
- mutation 自动 rollback。
- 启动时恢复未完成状态。

### B3：备份与恢复 UI

- 设置页列表、占用、打开位置、永久清理。
- restore plan、预览、执行和重新索引。

### B4：数据库与混合来源

- SQLite online backup。
- row graph restore。
- 多 artifact 顺序和冲突处理。

真实 edit/delete/convert mutation 只能在对应 backup 阶段可用后开放。

## 测试重点

- Windows、两个 WSL distro 分别使用独立 1 GiB 配额。
- WSL 离线不影响 Windows 清理。
- 单次备份超过 1 GiB 阻止 mutation；临时上限仅本次有效。
- 临时超额备份在 mutation 完成后按默认规则自动清理，不长期豁免。
- shared JSONL 恢复不覆盖 mutation 后追加的其他会话。
- WAL 数据库快照包含已提交数据。
- 主会话树删除中途失败后完整恢复。
- 应用崩溃后识别未完成 manifest 并恢复。
- 目标工具运行时备份后的 apply/restore 均被阻止。
- 清理不会删除执行中、恢复中和异常 manifest。
- 手动恢复成功后 History Index DB 与外部源一致。
