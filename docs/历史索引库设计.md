# 历史索引库设计

## 结论

继续使用现有 `history-catalog.db`，在同一个文件中建设 v2 schema，不新增第二个索引数据库。

History Index DB 是可重建的本地派生索引：

- 外部 Claude、Codex、Gemini、OpenCode 等历史文件/数据库是真实来源。
- 列表、搜索、统计和规范化详情读取 History Index DB。
- raw view、精确 diff 还原、编辑、删除和转换执行阶段按 `rawPointers` 回源。
- 编辑、删除、转换不能只修改索引库；先修改外部源，成功后重新解析受影响会话。
- 索引库不加密，不参与同步，依赖操作系统文件权限保护。

## 当前实现基线

当前 `src-tauri/src/commands/history/catalog.rs` 已具备：

- `history-catalog.db`。
- WAL、`synchronous=NORMAL`、foreign keys、5 秒 busy timeout。
- `history_catalog_sessions`、`history_catalog_messages`、FTS5 和 `history_catalog_state`。
- 文件 mtime/size/parser version 增量判断。
- 单文件替换事务和后台刷新状态事件。

当前限制：

- 主键是 `(roots_key, file_path)`，无法表达数据库 row、混合来源和同一工具的多个运行环境。
- `roots_key` 由 Claude/Codex 两个目录拼接，不是稳定来源实例。
- 消息只保存展示文本，tool call、tool result、thinking、附件等结构信息会丢失，不能支撑可靠跨格式转换。
- 详情、usage、tool events、file changes 仍需要回读原文件。
- 列表/搜索请求会通过 10 秒 TTL 触发目录扫描，不符合“读请求只查索引库”的目标。
- 扫描结果为空时会把旧路径当作 stale 删除；目录离线、WSL 未启动或权限失败时存在误清理风险。
- 同时维护 SQLite catalog 和 `history-index-cache.json`，存在双索引一致性问题。

## 数据边界

```text
来源设置 / 自动探测
  -> HistorySourceInstance
  -> adapter.discover() 返回轻量 sessionRef + fingerprint
  -> adapter.parse() 返回统一中间模型
  -> History Index DB
  -> list / search / stats / detail / conversion plan

mutation plan
  -> 备份外部目标
  -> adapter.apply()
  -> adapter.parse(affected session)
  -> History Index DB 增量替换
```

数据库内只保存三类数据：

1. 可从外部历史源重建的规范化内容。
2. 增量同步所需的 fingerprint、parser version、generation 和错误状态。
3. 临时同步运行记录。

以下数据不能只存在这个可重建库中：

- 用户自定义来源设置：继续由 settings store 保存。
- mutation 备份和恢复清单：保存在来源环境的备份目录。
- 未来的用户标签、备注等 CLI-Manager 私有数据：应进入持久业务库或单独的不可重建表域。

现有私有数据迁移要求：

- `session_meta`、`session_favorite_snapshots`、`history_edit_audit` 和前端 `makeSessionKey` 不能继续以裸 `file_path` 作为长期身份。
- 升级时为旧 Claude/Codex 记录建立 `(sourceId, sourceInstanceId, sessionId)` 映射，迁移标签、收藏、审计和置顶等私有元数据。
- 物理删除成功后，必须清理对应收藏快照和审计正文；保留必要审计摘要时不能继续保存已删除会话正文。
- private metadata 与可重建索引分表或明确命名空间，重建 History Index DB 不得丢失用户私有标记。

## 身份模型

### Source 类型

`sourceId` 是固定短 id，例如 `claude`、`codex`、`gemini`、`opencode`。

### Source 实例

同一种工具可能在不同位置存在多个候选，或随时间切换读取位置：

- Windows 本机 Claude。
- WSL Ubuntu Claude。
- WSL Debian Claude。
- 用户添加的另一个 Claude 根目录。
- 多个 OpenCode SQLite 数据库。

产品规则是每个 source 同一时间只有一个 active instance，但数据库仍需要保留 instance 身份以处理路径切换和迁移。因此会话主身份必须是：

```text
(source_id, source_instance_id, source_session_id)
```

规则：

- 用户确认读取位置时生成 instance id 并持久化到设置。
- 自动探测只返回候选，不自动创建 active instance。
- WSL 的 `environmentKey` 必须包含 distro。
- `source_session_id` 由 adapter 生成，优先使用来源原生 id。
- 如果来源没有原生 id，adapter 必须定义版本化稳定规则；通用层不能擅自用文件路径充当 session id。
- 路径、数据库路径和 row key 是定位字段，不是跨实例主键。

## v2 表结构

表名前缀统一使用 `history_`。v1 的 `history_catalog_*` 表在迁移期保留。

### 1. Schema 元数据

```sql
CREATE TABLE history_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at INTEGER NOT NULL
);
```

用途：记录 active schema、统一模型版本、最后重建时间。SQLite migration 仍以 `PRAGMA user_version` 为准。

### 2. 来源实例

```sql
CREATE TABLE history_source_instances (
  id TEXT PRIMARY KEY,
  source_id TEXT NOT NULL,
  environment_kind TEXT NOT NULL,
  environment_key TEXT NOT NULL,
  storage_kind TEXT NOT NULL,
  display_name TEXT,
  locations_json TEXT NOT NULL,
  settings_hash TEXT NOT NULL,
  activation_state TEXT NOT NULL DEFAULT 'pending',
  discovered INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE INDEX idx_history_source_instances_source
  ON history_source_instances(source_id, activation_state);

CREATE UNIQUE INDEX idx_history_source_instances_one_active
  ON history_source_instances(source_id)
  WHERE activation_state = 'active';
```

说明：

- `locations_json` 是设置/探测结果快照，不是设置的真实存储。
- `environment_kind` 为 `windows | wsl | macos | linux`。
- `environment_key` 对 WSL 是 distro，对原生环境是稳定系统标识。
- `activation_state` 使用 `pending | active | inactive`；partial unique index 保证一个 source 同时最多一个 active instance。

### 3. 会话

```sql

  timestamp_quality TEXT NOT NULL DEFAULT 'reported',
  message_count INTEGER NOT NULL DEFAULT 0,

  input_tokens INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL DEFAULT 0,
  cache_read_tokens INTEGER NOT NULL DEFAULT 0,
  cache_creation_tokens INTEGER NOT NULL DEFAULT 0,
  total_cost_usd REAL NOT NULL DEFAULT 0,
  usage_quality TEXT NOT NULL DEFAULT 'unknown',
  cost_kind TEXT NOT NULL DEFAULT 'unknown',
  pricing_version TEXT,
  dominant_model TEXT,
  current_model TEXT,
  context_window INTEGER,
  last_context_tokens INTEGER,
  reasoning_effort TEXT,
  tool_call_count INTEGER NOT NULL DEFAULT 0,

  fingerprint_kind TEXT NOT NULL,
  fingerprint_value TEXT NOT NULL,
  parser_version INTEGER NOT NULL,
  model_version INTEGER NOT NULL,
  parse_status TEXT NOT NULL,
  completeness_json TEXT,
  raw_pointers_json TEXT,
  source_extension_json TEXT,
  last_seen_generation INTEGER NOT NULL,
  indexed_at INTEGER NOT NULL,

    REFERENCES history_source_instances(id) ON DELETE CASCADE,

);

CREATE INDEX idx_history_sessions_updated
  ON history_sessions(source_instance_id, updated_at DESC, id);

CREATE INDEX idx_history_sessions_project
  ON history_sessions(cwd_normalized, updated_at DESC, id);

CREATE INDEX idx_history_sessions_created
  ON history_sessions(created_at DESC, id);
```

设计说明：

- child 表使用整数 `session_id` 外键，避免在每张表重复三列字符串主键。
- `fingerprint_value` 存 adapter 生成的 canonical fingerprint；通用库只做相等比较。
- `model_version` 是统一中间模型版本，和 adapter 的 `parser_version` 分开。
- `completeness_json` 记录 messages、usage、tools、fileChanges 等字段是否完整，转换前必须检查。
- `source_extension_json` 保存命名空间化来源私有字段，避免解析阶段不可逆丢失。
- `lifecycle_state` 区分 `active | archived | imported | unknown`，归档文件不能被误判为另一条会话。
- `timestamp_quality`、`usage_quality`、`cost_kind` 和 `pricing_version` 记录统计口径来源，避免把来源上报值、估算值和未知值混为一谈。

### 3.1 会话制品与关系

一条逻辑会话不等于一个文件。Claude 可能有主 transcript 和 subagents；Codex 可能有 rollout、`history.jsonl`、`session_index.jsonl` 和 state DB row；Cursor 可能跨多个数据库和 transcript。

```sql
CREATE TABLE history_session_artifacts (
  id INTEGER PRIMARY KEY,
  session_id INTEGER NOT NULL,
  artifact_index INTEGER NOT NULL,
  role TEXT NOT NULL,
  kind TEXT NOT NULL,
  ownership TEXT NOT NULL,
  locator_json TEXT NOT NULL,
  fingerprint_kind TEXT,
  fingerprint_value TEXT,
  writable INTEGER NOT NULL DEFAULT 0,
  source_schema_version TEXT,
  FOREIGN KEY(session_id) REFERENCES history_sessions(id) ON DELETE CASCADE,
  UNIQUE(session_id, artifact_index)
);

CREATE TABLE history_session_relations (
  parent_session_id INTEGER NOT NULL,
  child_session_id INTEGER NOT NULL,
  relation_kind TEXT NOT NULL,
  relation_index INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY(parent_session_id, child_session_id, relation_kind),
  FOREIGN KEY(parent_session_id) REFERENCES history_sessions(id) ON DELETE CASCADE,
  FOREIGN KEY(child_session_id) REFERENCES history_sessions(id) ON DELETE CASCADE
);
```

规则：

- `role` 使用 `primary | registry | metadata | subtask | attachment | auxiliary`。
- `ownership` 使用 `exclusive | shared`。`history.jsonl`、`session_index.jsonl`、state DB 等共享制品只能重写目标 record/row，不能因删除一条会话而删除整个制品。
- adapter 必须枚举完整 artifact graph，物理删除、恢复和目标写入不能只处理 primary file。
- 同一原生 session id 的 active/archive 表示由 adapter 合并为一个逻辑会话，并通过 lifecycle/artifact 表达状态。
- parent/subagent、fork、continuation 使用关系表，不把子会话消息永久压平到父会话。
- relation 支持任意深度，`relation_index` 保存同级稳定顺序；adapter 必须检测循环和悬空 parent 并报告 parse warning。
- 现有前端通过 `file_path/subagents/agent-*.jsonl` 推断一层下挂；v2 切换后由显式 relation 驱动，同时保持当前列表展开/收起效果。
- 现有 `aggregateSubtasks` 按时间压平消息主要用于汇总统计；索引库同时保留每条逻辑会话和关系树，不能只保存聚合结果。

### 4. 消息与消息部件

```sql
  cache_creation_tokens INTEGER,

  raw_pointers_json TEXT,
  source_extension_json TEXT,

CREATE TABLE history_message_parts (
  id INTEGER PRIMARY KEY,
  message_id INTEGER NOT NULL,
  part_index INTEGER NOT NULL,
  kind TEXT NOT NULL,
  text_content TEXT,
  mime_type TEXT,
  tool_call_id TEXT,
  tool_name TEXT,
  content_storage TEXT NOT NULL DEFAULT 'inline',
  payload_json TEXT,

  source_extension_json TEXT,

  UNIQUE(message_id, part_index)

```

`kind` 至少支持：

- `text`
- `thinking`
- `tool_call`
- `tool_result`
- `image`
- `file`
- `unknown`

`display_content` 服务现有 UI 和全文搜索；转换器使用 `history_message_parts`，不能从扁平展示文本反推协议结构。

超大 tool payload、二进制附件和图片不强制完整内联。message part 需要标记 `inline | rawOnly | unavailable`：

- `inline`：索引中包含转换所需内容。
- `rawOnly`：索引只保存摘要和 raw pointer，转换执行时按 fingerprint 回源。
- `unavailable`：来源已经无法提供完整内容，转换 plan 必须报告损失。

## 增量同步算法

### 发现阶段

adapter 返回：

```rust
struct HistoryDiscoveryResult {
    complete: bool,
    cursor: Option<String>,
    batch: Vec<DiscoveredSession>,
    warnings: Vec<HistorySourceWarning>,
}

struct DiscoveredSession {
    session_ref: HistorySessionRef,
    discovery_key: String,
    fingerprint_kind: String,
    fingerprint_value: String,
}
```

`complete=false` 表示目录离线、WSL 未启动、数据库锁/权限问题或扫描范围不完整。大来源不能一次性返回完整 `Vec`；adapter 必须支持分页或流式 discovery，通用同步器把 batch 写入 staging 后再继续拉取。

### 刷新步骤

1. 唯一同步协调器为单个 `sourceInstanceId` 分配新 generation，写入 `history_sync_runs`。不能同时运行 catalog 旧扫描、legacy 旧索引和 v2 shadow 扫描三套循环。
2. 在事务外执行 adapter discovery，按 batch 写入 staging 表。
3. 只对新增、删除、fingerprint 变化或 parser/model version 变化的会话更新 generation；无变化会话不能每轮写 `last_seen_generation`，避免 O(N) WAL 写放大。
4. 比较 fingerprint、parser version、model version，只解析新增或变化会话。
5. 解析在 SQLite 写事务外并行执行。
6. 每个变化会话用一个短事务替换 session/messages/parts/events/usage/fileChanges/FTS；Claude/Codex append-only JSONL 必须保存 parse cursor、offset 和尾部 hash，新增一条消息不能全量重建大 transcript。
7. 解析失败时保留旧索引内容，并写入 `history_index_failures`；新会话失败则只记录 failure。
8. 只有 discovery `complete=true` 时，才删除 `last_seen_generation < current_generation` 的派生会话。
9. 完成后更新 source state，并发出一次 generation 变化事件。

同一逻辑会话的多个 artifact 必须合成一个 fingerprint；任一制品变化都触发该会话重新解析。来源 schema 或 CLI 版本变化不直接等于可写，必须由 adapter 版本闸门判断。

禁止行为：

- 读 API 触发同步全量扫描。
- 某个来源扫描失败时清理该来源旧索引。
- 一个来源失败阻止其他来源刷新。
- 在长时间解析期间持有 SQLite 写事务。

## 刷新触发

读请求只查库。刷新由以下事件触发：

- 应用启动后的低优先级后台刷新。
- 用户点击“刷新”。
- 来源启停、目录或数据库位置变化。
- mutation 成功后定向刷新受影响会话。
- 后续文件监听器 debounce 后触发对应来源实例刷新。
- 后台周期检查，作为监听遗漏的兜底。

第一阶段不要求所有来源实现文件监听。保留手动刷新和后台周期检查即可，但必须移除“列表/搜索请求每 10 秒触发扫描”的耦合。

## Active instance 两阶段切换

用户更换来源读取位置时，不能先覆盖旧设置再尝试扫描。切换流程：

1. 新位置作为 pending instance 保存到临时设置状态，旧 active instance 继续服务列表、搜索、统计和详情。
2. 校验路径类型、环境、权限、来源格式/schema 和 canonical locator。
3. 为 pending instance 建立隔离的首轮索引；查询不能混入 pending 数据。
4. 首轮 discovery 必须 `complete=true`，关键 parser 校验通过后，在一个短事务中把旧 instance 设为 inactive、新 instance 设为 active。
5. settings store 只在数据库切换成功后提交新 active instance。
6. 切换后清理旧 instance 的派生索引；清理失败只记录维护错误，不回退已成功的新实例。

失败与取消：

- 新位置验证、扫描、解析或切换失败时删除 pending 派生数据，继续使用旧 active instance。
- 用户取消时不改变旧设置和历史列表。
- pending 阶段禁用该来源的 edit/delete/convertTo mutation，避免用户误以为操作目标已经切换；只读查询继续使用旧实例。
- 新旧 canonical locator 相同视为设置更新，不创建新的 instance id 或重建全库，除非 parser/settings hash 变化。
- UI 展示 `validating | indexing | switching | failed` 状态和失败原因，不能在切换完成前显示“已保存”。

## 查询路径

### 列表

`history_sessions + history_source_instances`，按 `(updated_at DESC, id DESC)` keyset pagination。不要长期使用大 offset。

### 搜索

`history_messages_fts -> history_messages -> history_sessions -> history_source_instances`。过滤 source、instance、project、时间范围后返回 message snippet。

### 详情

详情拆成 header、message keyset page、tool/diff/usage 懒加载。不能把完整会话一次性经 IPC 传到前端再只依赖虚拟列表；前端虚拟化不能抵消 IPC、序列化和 JS 内存成本。

### 统计

- 会话数和总 token：`history_sessions` 聚合列。
- 模型分布：`history_session_model_usage`。
- 每日/小时趋势：`history_usage_events`；缺少 usage event 的来源回退 session created/updated time。
- 工具统计：`history_tool_events`。
- subagent 另外计算：每个节点保存自身指标；全局消息/token 只统计一次；session 数默认只计主会话，subagent 数单列展示。
- 日期、小时、来源、项目等聚合只返回 count/token/cost 等 aggregate，不在桶内返回全部 `session_refs`；下钻列表单独分页查询。
- 列表分页按 root session 计算，child 不占 root 页；展开时批量懒加载关系树，避免 N+1。

## 外部数据库来源

OpenCode、Cursor 等外部 SQLite/混合存储只能通过 adapter 访问。Kiro 当前版本是 workspace-sessions JSON，不进入本节数据库规则：

- 解析阶段使用只读连接和短读事务。
- 不使用 `immutable=1` 打开仍可能存在 WAL 写入的活动数据库。
- busy/locked 时返回来源暂不可用，保留现有索引。
- 第一阶段禁止写外部数据库。
- 数据库/混合写入不设置通用实验性开关；只有 source-specific writer 完成 schema/version 验证、Backup Service 能力、运行时锁检测和恢复测试后，才允许声明 `convertTo/edit/delete=supported`。
- adapter 必须探测来源 schema/version。未知版本允许按已验证的兼容策略只读解析，但默认关闭 edit/delete/convertTo。
- 数据库写入前检查源应用是否仍在运行、锁状态和 schema fingerprint；需要关闭应用时必须明确提示，不能强行结束进程。
- 对转换目标，检测到目标工具正在运行时一律禁止写入，即使数据库暂时未返回 locked；capability 或用户确认也不能绕过该限制。

## Mutation 与备份

备份与恢复已经拆为独立设计和 Trellis 子任务，完整方案见 [`docs/历史备份与恢复设计.md`](./历史备份与恢复设计.md)。本节仅保留 History Index DB 与 mutation executor 的集成约束；如有冲突，以独立备份设计为准。

这里的备份是“修改外部原始历史前的恢复快照”，不是 History Index DB 备份。

### 需要备份

- 编辑：备份将被重写的源文件或数据库。
- 物理删除：备份包含目标会话的源文件/数据库状态。
- 转换：转换固定不修改来源，因此不备份来源；如果目标需要更新已有 registry/database 或其他共享制品，只备份会被修改的目标。

### 备份内容

文件型来源：

- 第一阶段复制完整受影响文件，避免只备份 JSONL 行后无法可靠恢复协议关系。
- 保存原路径、mtime、size、hash、sessionRef、rawPointers。

SQLite 来源：

- 允许已验证 shared artifact 使用逻辑备份：append-only JSONL 保存截断点和尾部 hash，SQLite 保存受影响 row graph、外键边界和 reverse operation。
- 活动 WAL 数据库使用 SQLite online backup、事务化 row graph 备份，或 adapter 验证过的一致性快照方式；不能只复制主 `.db` 文件。
- 未知 schema、独占文件、破坏性重写、跨版本未验证或无法证明 reverse operation 完整时，必须使用完整一致性快照。

混合来源：

- 同一个 mutation id 下同时保存文件和数据库快照。
- manifest 记录恢复顺序和一致性 hash。

建议目录结构：

```text
~/.cli-manager/backups/
  <source-id>/
    <source-instance-id>/
      <yyyyMMdd-HHmmss>-<mutation-id>/
        manifest.json
        files/...
        database/...
```

备份保持来源原始格式，不加密。每个运行环境的 backup root 独立执行配额：默认保留 7 天或最多占用 1 GiB，任一条件先达到即触发清理；同时提供按 mutation 和该环境全部备份的“立即永久清理”。

自动清理规则：

- 只清理已经完成且不处于恢复中的备份目录。
- 每个运行环境分别统计。先删除该 backup root 中超过 7 天的备份，再按创建时间从旧到新清理，直到该 root 总占用不超过 1 GiB。
- 正在执行、恢复中、manifest 不完整的目录不自动删除，标记为异常并提示用户处理。
- 清理以完整 mutation 目录为单位，不能只删除其中的数据库或文件快照。
- 清理失败不阻断历史读取，但必须在设置页显示失败原因和当前占用。

已确认：1 GiB 按每个运行环境的 backup root 独立计算，不跨 Windows、不同 WSL distro、macOS/Linux 做全局合计。某个环境离线不影响其他环境的统计和清理。

已确认：如果单次必要备份预计就超过该运行环境的 1 GiB 上限，必须阻止 mutation，并提示用户选择其他备份目录或显式临时提高该环境上限。不能为了满足限额跳过备份，也不能静默突破上限。

临时上限规则：

- 只作用于当前运行环境，不改变其他 backup root。
- 必须由用户显式输入或确认，并在 mutation plan 中展示预计备份大小和临时上限。
- 只对当前 mutation 有效；完成或取消后恢复默认 1 GiB。
- 临时超额备份不享受长期豁免；mutation 执行中、回滚中和恢复中禁止清理，完成后立即按默认 7 天 / 1 GiB 自动清理。
- 如果完成后该环境仍超过默认 1 GiB，清理会先删旧备份；仍无法降到上限时，本次已完成备份也可被自动删除。确认弹窗必须明确提示这一点。

## Mutation 执行与恢复

现有 Claude/Codex 转换会依次创建 rollout、追加 JSONL index、更新 state DB；任一步失败都会产生半完成状态。通用 mutation executor 必须提供状态机：

```text
planned
  -> preflighted
  -> backingUp
  -> staged
  -> applying
  -> verifying
  -> completed

任一步失败 -> rollingBack -> rolledBack | manualRecoveryRequired
```

要求：

- 每个 mutation 有稳定 `mutationId` 和 idempotency key，重试不能重复追加 registry row 或生成第二份目标会话。
- plan 为每个 artifact 保存执行前 fingerprint、预计执行后 fingerprint、ownership 和精确 operation；apply 前任一前置条件变化都必须中止并重新生成 plan。
- mutation workspace 与 `manifest.json` 在执行前创建，状态更新采用临时文件 + atomic rename。
- 文件 writer 先写同目录临时文件、flush，再 atomic rename；不能直接 `File::create` 覆盖最终文件。
- SQLite 内部操作使用事务；跨文件和数据库无法共享事务，必须按 plan 定义提交顺序和逆操作。
- 通常先落地可独立验证的主会话制品，再更新 registry/index；失败时根据备份恢复已修改目标并删除新建制品。
- 应用启动时扫描未完成 manifest，允许继续验证、自动回滚或进入人工恢复，不能静默忽略。
- 备份必须可恢复。adapter 需要 `build_restore_plan` 和 `apply_restore_plan`，恢复前同样检查目标 fingerprint，避免覆盖用户之后的新修改。
- 共享文件/数据库的完整快照主要用于灾难恢复；正常恢复优先执行逆向 record/row operation。mutation 后来源又产生新数据时，不能用旧整库/整文件快照覆盖无关变化。
- mutation 完成不以“写入返回成功”为准，必须重新解析目标会话并校验 session id、消息数、关键结构和 writer 版本。

删除级联规则：

- 删除主会话的 mutation scope 固定包含全部 subagent 后代和各自 artifact graph。
- plan 按树展示主会话、后代数量、共享/独占制品和预计备份大小。
- 整棵树的路径、fingerprint、权限、空间和备份 preflight 全部成功后，才允许执行第一步物理删除。
- 删除 shared artifact 时只移除对应 record/row；exclusive artifact 才允许删除整个文件或目录。
- 删除完成后重新扫描来源，主会话及全部后代都不得继续出现在索引；来源离线时进入待校验状态，不能伪报完成。
- 已确认全有或全无语义：任一 subagent 或 artifact 删除失败，立即停止后续删除，并从 mutation 备份恢复此前已经修改/删除的主会话、subagent 和共享 record/row。
- 回滚并重新解析验证成功后，mutation 状态为 `failedRolledBack`；不存在“部分删除成功”。
- 回滚失败时状态为 `manualRecoveryRequired`，锁定该来源实例的后续 mutation，只保留读取、导出备份和人工恢复入口。
- History Index DB 只在整棵树删除验证成功后移除会话；失败回滚后重新解析恢复原索引状态。

subagent 操作规则：

- subagent 是只读关联节点，不允许作为 edit、delete 或 convert mutation root。
- subagent 的消息即使 raw pointer 可写，索引中的 `editable` 也必须为 false；主会话聚合详情和单独 subagent 详情保持一致。
- UI 禁用编辑、删除、转换入口并说明只能从主会话操作。
- 后端分别返回 `history_edit_subagent_not_allowed`、`history_delete_subagent_not_allowed`、`history_conversion_subagent_not_allowed`。
- 批量操作只要把 subagent 当作独立 root 提交就拒绝；主会话 delete plan 内部展开出的 subagent 不受此限制。
- orphan subagent 同样只读，必须先恢复 parent relation，不能通过缺少 parent 绕过限制。

转换 scope 对存在 parent/subagent/fork 的会话必须显式记录：选择主会话时，conversion plan 检查主会话及其全部后代关系。目标 writer 支持关系时按树写入；目标格式不能表达关系，或某个 subagent 无法安全写入时，该 subagent 不生成独立目标会话，也不压平进主会话，而是作为转换丢弃项列出并要求用户确认。

转换丢弃规则：

- “丢弃”只表示不写入目标，绝不删除或修改来源 subagent。
- plan 的 `droppedSubagents[]` 包含 source session ref、关系路径、原因、消息数、tool/event 数和受影响后代数。
- subagent 转换失败时丢弃该 subagent 及无法重新挂接的整个后代子树；其他可保留关系的 subagent 可以继续。
- 主会话写入失败必须使整个 mutation 失败并回滚，不能用“有损转换”绕过主会话错误。
- 执行结果必须再次显示成功转换和丢弃的 subagent 数量，不能只在确认弹窗中出现一次。
- 已确认部分成功语义：例如 5 个 subagent 中第 3 个失败，主会话和其余 4 个继续转换，仅丢弃第 3 个及其无法挂接的后代。
- 每个 subagent 子树是独立 apply/verify/compensate 单元；某单元失败时回滚该单元已经创建的目标制品，不回滚已成功主会话和其他 subagent。
- 用户确认弹窗必须明确说明“运行期间失败的 subagent 将被跳过并继续”，执行结果状态使用 `completedWithOmissions`，无丢弃时才是 `completed`。
- 成功关系映射只包含实际写入并通过目标重解析验证的节点；失败节点不能残留 registry、database row 或孤立目标文件。

转换入口规则：

- subagent 不能作为 conversion root，前端单独选中 subagent 时禁用转换入口并提示“请从主会话发起转换”。
- 后端必须根据显式 `session relation` 再次校验并返回稳定错误 `history_conversion_subagent_not_allowed`，不能只依赖前端隐藏或路径判断。
- orphan subagent 即使暂时找不到 parent，也不允许独立转换；应先修复/刷新关系或从可识别主会话发起。
- source descriptor 的 `convertFrom` 只是来源级能力，实际入口还要与 session-level constraints 取交集。

## 路径、重复来源与安全边界

- 自动探测候选和用户输入都要做 canonicalize、大小写/分隔符归一化和 WSL distro 识别。
- 同一 source 只保存一个 active instance；多槽位来源内部的重叠根目录由 adapter ownership 规则去重。
- 默认不递归跟随 symlink/junction；显式选择的符号链接只解析一次真实目标，并检测循环。
- 扫描器必须排除 `.cli-manager/backups`、History Index DB、临时 mutation workspace 和来源自身 cache，防止把备份重新索引。
- raw pointer 和 mutation target 必须再次验证仍位于 adapter 声明的 artifact graph 内，防止路径穿越或过期指针修改无关文件。
- WSL 来源在 distro 不可用、Linux home 无法写入备份或权限不匹配时禁止 mutation，但保留现有索引可读。

## 格式版本与转换契约

capability 不是只按 source id 判断，而是按 `sourceId + detectedFormatVersion + adapterVersion` 判断：

- reader 声明已验证读取版本范围。
- writer 声明已验证写入版本范围和目标原生 schema。
- 未知版本默认只读或 planned，不能沿用旧 writer 猜测写入。
- conversion plan 记录 reader/writer/model 版本，执行时版本变化必须重新生成 plan。
- 每个 writer 必须有 golden fixture、目标工具可发现性验证和 parser -> writer -> parser 语义 round-trip 测试。
- `convertTo` 与 `resume` 分开；成功写出格式不等于目标工具一定能 resume，兼容性报告必须分别说明。

## Writer 与目标会话包

writer 的交付单位不是单个 transcript，而是目标工具能够识别、展示和 resume 的完整目标会话包。

通用规则：

- 每次转换生成新的目标原生 session id，不复用来源 id，不覆盖、不合并已有会话。
- plan 在任何写入前检查新 id 是否出现在目标任一文件、registry 或数据库中；无关碰撞则重新生成 id。
- 同一个 `mutationId/idempotencyKey` 重试时复用第一次分配的目标 id 和 artifact plan，不能再创建一条新会话。
- writer 声明 `requiredArtifacts`、`sharedArtifacts`、写入顺序、验证方式和逆操作。
- `convertTo=supported` 只在完整会话包 writer 与目标重解析验证都通过后开放；只会写主文件的 adapter 仍是 `planned`。
- 缺少目标版本的必需 artifact、schema 未知或 writer 未验证时，conversion preflight 直接失败，禁止 best-effort 写入。
- 目标会话包不完整是结构性不可执行错误，不属于可由用户确认绕过的“有损转换”。
- preflight 失败时不得创建 rollout/transcript、不得追加 registry，也不得修改数据库；UI 只展示修复建议，例如先启动目标工具完成初始化或选择受支持版本。
- 目标工具处于运行状态时 conversion preflight 必须失败，不允许用户强制继续，也不允许 CLI-Manager 强制结束目标进程。
- adapter 负责检测当前环境中的目标进程和活动写锁；WSL 目标必须在对应 distro 内检测，不能只检查 Windows 进程表。
- process/lock 检查在 plan 预览前和实际 commit 前各执行一次；两次之间目标状态变化时中止并要求重新生成 plan。
- 关闭目标工具后用户需要重新 preflight，旧 plan 不能直接复用。

Codex 当前实现已经说明为什么不能只写索引：

- rollout JSONL：真正的会话 transcript，文件名和 `session_meta` 使用同一个新 UUID。
- `history.jsonl`：共享历史入口，需要追加同一个 `session_id`。
- `session_index.jsonl`：共享会话 registry，需要写 id、标题、cwd、rollout path。
- `state_5.sqlite`：`threads` row 保存 thread id、rollout path、model、cwd、preview、时间等状态。

Codex target bundle 示例：

```text
TargetSessionBundle(codex, newSessionId)
  ├─ create rollout-...-<id>.jsonl          [exclusive]
  ├─ append/update history.jsonl record      [shared]
  ├─ append/update session_index.jsonl row   [shared]
  └─ insert threads row in state_5.sqlite    [shared database]
```

执行要求：

1. 检测 Codex 目标格式/schema 版本，确定该版本的必需制品，不能把上述固定列表盲目套到所有版本。
2. 为 shared JSONL 和 state DB 创建一致性备份，rollout 在目标目录 staging。
3. 所有 artifact 使用同一个新 session id 和互相一致的 path/cwd/timestamp。
4. 按 plan 提交；任一步失败时恢复 shared artifact 并删除新 rollout。
5. 重新通过 Codex adapter 解析目标，检查 thread row、rollout、registry、消息数和 resume capability。

若检测到的 Codex 版本要求 `state_5.sqlite`/thread registry，但数据库不存在或 `threads` schema 不受支持，则返回 `history_target_bundle_incomplete` 或 `history_target_schema_unsupported`，不允许沿用现有“跳过 thread registration 仍算成功”的行为。

其他来源同样需要 source-specific bundle：OpenCode 是 session/message/part row graph，Cursor 可能跨 transcript 与多个 `state.vscdb`，Claude 可能涉及 transcript、session index 和 subagent 关系。不能用一个通用 JSONL writer 冒充原生转换。

## 容量、维护和失败降级

- History Index DB 不设置按时间淘汰正文的默认上限，否则列表与搜索会出现难以解释的缺口。
- 设置页展示索引库大小、FTS 大小、最后成功同步、失败会话数和各来源占用。
- 磁盘不足时停止新索引写入，保留旧索引可读并提示释放空间；不能删除外部源或静默清理最近会话。
- FTS optimize、WAL checkpoint 和 vacuum 放在低频维护任务，不在每次会话写入后执行。
- 单条超大内容使用 rawOnly 策略，解析器使用流式读取和有界并发，防止大 JSONL/数据库 blob 导致 OOM。

## 有损转换

用户已确认：允许有损转换，但必须提醒并由用户明确确认。

转换是非破坏性复制：

- 成功后来源会话、subagent、文件和数据库 row 全部保持不变。
- 转换过程中只读取来源；fingerprint 变化时中止并重新生成 plan，不能写回来源。
- 不提供“转换后自动删除来源”或隐式 move。
- 用户需要清理来源时，必须在转换完成后单独发起物理删除，重新执行 plan、备份和确认。
- 目标写入失败或回滚不影响来源；转换结果保留 source ref 与 target ref 的映射用于追溯。
- 目标会话 id 永远新建；不覆盖、不合并现有会话。若 id 在目标会话包任一制品中碰撞，写入前重新生成。

conversion plan 必须包含：

- `lossless: boolean`
- `losses[]`：字段、数量、示例、原因、影响等级。
- 目标来源 writer 版本。
- 将创建/覆盖/修改的目标文件或数据库 row。

未确认前只能预览或导出 plan，不能执行。有损项至少覆盖 thinking、tool call/result、usage、diff、附件、来源私有元数据和 resume 能力变化。

## 明文存储

用户已确认不加密：

- `history-catalog.db` 使用普通 SQLite。
- 不引入 SQLCipher 或新的加密依赖。
- 索引库和备份不参与云同步。
- 目录权限使用当前用户最小可读写权限。
- UI 明确说明索引库和备份包含会话正文。

物理删除会清理索引记录和外部源记录，但 SQLite/SSD 层面的法证级不可恢复不作承诺。执行“永久清理备份”时应同时删除对应备份，并对索引执行必要的 FTS 清理和 WAL checkpoint。

## 迁移方案

1. 使用 `PRAGMA user_version` 增加 v2 migration，在现有 DB 内创建 `history_*` 表。
2. H1 只迁移设置、descriptor 和路径模型，旧历史查询链路不变。
3. H2 v1 `history_catalog_*` 继续服务查询，后台从外部源 shadow build v2，不能直接把 `file_path` 当成新主键迁移。
4. 对比来源数、主会话数、subagent 数、消息数、抽样详情和 FTS 搜索结果。
5. H3 验证通过后切换 list/search。
6. H4 切换分页 detail、subagent、prompt、diff。
7. H5 最后切换 stats，并验证统计 payload 不随会话数线性增长。
8. 稳定一个版本后停止读取和写入 `history-index-cache.json`。
9. 再稳定一个版本后删除 v1 表和旧 JSON 缓存清理逻辑。

索引损坏时允许重建，但不直接静默删除：先把损坏文件重命名为带时间戳的 `.corrupt` 文件，创建新库并向 UI 报告重建状态。

## 性能目标

- 已完成索引时，列表首屏查询不触发源扫描。
- 10 万会话规模下，列表首屏和普通筛选目标小于 200 ms。
- 100 万消息规模下，常规全文搜索目标小于 500 ms，不包含首次冷缓存。
- 单来源刷新失败不影响其他来源查询。
- 内存中不长期保留完整会话副本；批量解析有明确并发和批次上限。
- 同样本 Claude/Codex P95 相比当前版本回退不超过 10%。
- 后台同步期间，list/search/detail/stats 前台 P95 回退不超过 20%。
- 无变化刷新不写 session/message/FTS，不更新 generation，不触发 UI 刷新。
- 读 API 零源扫描、零 WSL shell out；active session 只允许定向 fingerprint 检查。
- SQLite 启动时一次 schema 初始化；小连接池、单 writer queue、短事务、批量 FTS 构建。
- 重内容只保存摘要或 relative raw pointer，不重复保存完整 tool JSON、patch old/new text 和附件正文。
- 监控 IPC 字节数、normalize 耗时、React commit、RSS、DB/WAL 峰值和 WSL 进程次数。

这些是设计目标，实施前需要用现有大历史样本建立 benchmark，而不是直接当作已验收数据。

## 第一阶段最小范围

第一阶段只实现支撑多来源接入的核心：

- v2 schema migration。
- source instance、sessions、artifacts、relations、messages、message parts、FTS、source state、sync runs、failures。
- Claude/Codex adapter 输出统一模型并 shadow 重建 v2。
- list/search/detail/stats 一致性对比验证；H1/H2 不切实际读取路径。
- 安全 generation 删除规则。

usage events、tool events、file changes 可按现有页面切换顺序分批迁移，但在 detail 完全切库前必须补齐。mutation、备份执行和外部数据库写入不属于索引库第一阶段。

## 测试重点

- Windows + 两个 WSL distro 存在相同 session id，不互相覆盖。
- 同一来源不能同时启用多个目录/数据库；切换 active instance 后新旧索引不会混合展示。
- 来源目录离线、WSL 未启动、数据库 locked，不删除旧索引。
- parser version 升级只重建对应来源。
- 单会话解析失败保留旧可用内容并展示失败状态。
- message parts、tool calls、usage、diff 从源到索引再到详情不丢结构。
- FTS insert/update/delete 与消息表一致。
- mutation 成功后只刷新受影响会话。
- 索引库删除后可以从外部源完整重建。
- Codex 等多制品会话发生中途失败时可以回滚，重启后能发现未完成 mutation。
- 来源 format/schema 版本未知时不会错误开放 writer。
- 自定义目录重叠、符号链接和备份目录不会产生重复或循环索引。
- usage/cost 的 reported、estimated、unknown 口径不会混合成误导性统计。
